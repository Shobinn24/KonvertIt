KonvertIt Scrapers Audit

Scope

Reviewed files in app/scrapers/:
  • base_scraper.py
  • browser_manager.py
  • proxy_manager.py
  • scraper_factory.py
  • amazon_scraper.py
  • walmart_scraper.py
Objective: assess robustness, anti-bot strategies, correctness, performance, resilience, and test coverage alignment. Provide actionable improvements.

Summary

  • Strengths: clean template-method design; integrated CircuitBreaker + retry backoff; proxy health tracking with rotation; Playwright pool with stealth tweaks; ScraperAPI structured endpoints preferred; URL normalization; explicit bot-detection heuristics; graceful ProductNotFound/429 handling.
  • Medium risks: BaseHTTPMiddleware-like page waits may miss late-rendered content; structured endpoint dependency centralizes failure on ScraperAPI; proxy API key extraction from address string is brittle; direct fallback may hit bot walls quickly; no per-domain concurrency caps.
  • High risks: unbounded HTML size parsing could lead to excess memory usage on pathological pages; reliance on lxml/BeautifulSoup without size guard; potential misuse of ScraperAPI key if logs include full proxy address (ensure redaction upstream).

Detailed Findings

1) BaseScraper (template, resilience, validation)

Resilience
Wraps scrape in CircuitBreaker and retry_with_backoff (ScrapingError retryable). Proxy success/failure feeds health model. Browser resources released in finally.
Navigation
Supports ScraperAPI by URL wrapping; 404→ProductNotFound, 429→RateLimitError. Wait strategy via _wait_for_content (default fixed 2s).
Bot detection
Generic checks (captcha/robot/verify/access denied) and short-circuit via subclass override.
Validation
validate() uses ScrapedProduct.is_complete; raises ScrapingError if incomplete.
  • Good: resource hygiene, explicit error taxonomy, URL cleaning hook.
  • Consider streaming-compression awareness; page.content() returns full HTML; add max size guard.
  • Backoff parameters static; consider jitter and marketplace-specific policy.

2) BrowserManager (anti-detection, pooling)

Pool
Semaphore limits concurrent contexts to pool_size; per-call creates new context/page; releases on close; headless configurable.
Stealth
Random UA and viewport; webdriver flag removed; headers set. Proxy ignored for ScraperAPI (uses URL wrapping).
Delays
Human delay randomized between settings.scrape_min_delay and scrape_max_delay.
  • Good: per-page context reduces cross-contamination fingerprints.
  • Creating a new context per request is heavier; consider a small context pool if memory allows.
  • Missing per-domain concurrency cap and rate pacing beyond human delay.

3) ProxyManager (rotation, health)

Sources
Supports ScraperAPI (endpoint mode) and raw proxy list. Falls back to direct if empty or all inactive.
Health
Per-proxy success/failure with health_score; deactivates at 0; reactivate_all resets health.
Selection
Type-priority then health-sort with round-robin among active.
  • Good: simple, effective selection and deactivation strategy.
  • API key extraction from address uses string slicing; prefer parsing query params to avoid leakage and edge cases.
  • No cooldown/backoff per proxy after failures; repeated rapid retries can burn IP reputation.

4) AmazonScraper

Strategy
Structured endpoint preferred; fallback uses robust selectors and image upscaling; dog/CAPTCHA detection; ASIN normalization to canonical URL.
Parsing
Price regex handles various blocks; brand cleanup; bullet-to-description; image de-dup and high-res mapping.
Bot detection
Explicit Dog/CAPTCHA exceptions; very short page heuristic (<10KB).
  • Good: covers common Amazon variants; guards against tracking pixels; eBay 12-image cap.
  • Short-page threshold may produce false positives for localized minimal pages; consider combining with HTTP status and specific DOM checks.
  • BeautifulSoup + lxml parse on full DOM could be heavy; add upper size limit (e.g., 3–5 MB) and early abort.

5) WalmartScraper

Strategy
Structured endpoint default; PerimeterX handled by provider; fallback selectors provided.
Parsing
Multiple price sources (content attr, text); image normalization; categories from breadcrumbs.
Bot detection
PerimeterX and short-page heuristics; logs warning and returns True for generic blocks.
  • Good: pragmatic anti-bot handling with provider offload.
  • Like Amazon, add HTML size guard and stricter fallback waits for critical selectors.

6) ScraperFactory

Extensibility
Static registry with default registrations; clear error if unsupported.
  • Good: simple and explicit; supports DI of managers.
  • If hot-reloading, ensure imports do not double-register (idempotent guard exists implicitly via dict overwrite).

Actionable Recommendations

  1. Content size and safety:
    • Introduce a max HTML size guard before BeautifulSoup parsing (e.g., 3–5 MB) and log/raise ScrapingError if exceeded.
    • Consider stripping scripts/styles before parsing to reduce memory.
  2. ScraperAPI integration:
    • Parse ScraperAPI keys using urllib.parse to avoid brittle slicing; add redaction in logs.
    • Add circuit-breaker separation for structured endpoints vs Playwright path so correlated failures don’t trip both.
  3. Concurrency and pacing:
    • Add per-domain concurrency caps and a token-bucket rate pacer to respect site limits.
    • Add proxy cooldown after N consecutive failures; exponential backoff per proxy to protect reputation.
  4. Robust waits and detection:
    • Augment _wait_for_content in BaseScraper with marketplace-supplied selectors or a callback to wait for mission-critical elements.
    • Combine short-page heuristic with DOM checks (e.g., presence/absence of expected containers) to reduce false positives.
  5. Observability:
    • Emit metrics: scrape_duration, success/failure counts by source, proxy success_rate histogram, captcha/dog detections, structured vs fallback ratios.
    • Attach request_id (from middleware) to scraper logs via contextvars if scraping happens inside request scope.
  6. Testing:
    • Add unit tests for URL cleaning (Amazon/Walmart) and ID extraction edge cases.
    • Mock Playwright/page to test _wait_for_content timeouts and error paths.
    • Test ProxyManager reactivation and health decay, including direct fallback path.
    • Test structured endpoint error handling (404/429/5xx) and redaction of API keys in logs.

Configuration Assumptions

  • Playwright is installed and usable in the deployment environment; Chrome dependencies available.
  • Settings.browser_pool_size and scrape_min/max_delay tuned to throughput vs. block risk.
  • Proxy provider and keys are provided via env; secrets are not logged in plaintext.
