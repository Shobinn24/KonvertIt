KonvertIt Middleware Audit

Scope

Reviewed files in app/middleware/:
  • auth_middleware.py
  • exception_handler.py
  • logging_middleware.py
  • rate_limiter.py
  • security_headers.py
Objective: assess security, correctness, reliability, performance, observability, and test coverage alignment. Produce actionable recommendations.

Summary

  • Strengths: clear separation of concerns, robust JWT decode path, structured logging with request correlation, safe fail-open on Redis, minimal header surface, production-only HSTS.
  • Medium risks: BaseHTTPMiddleware limitations for long-running bodies; CSP overly strict could affect future features; Redis connection lifecycle singleton may leak across reloads; optional auth swallows errors broadly.
  • High risks: none identified in code review. Ensure secrets/algorithms align with configuration and token issuance.

Detailed Findings

1) Authentication (auth_middleware.py)

JWT validation
Uses jose.jwt.decode with Settings.secret_key and single algorithm from settings. Validates subject and token type for access flows. On JWTError → 401 with WWW-Authenticate.
Error handling
Logs warnings on verification failure; returns 401. Optional variant returns None on any HTTPException (suppresses specific causes).
User status
require_active_user validates UUID format, loads user from DB, checks is_active; returns 403 for deactivated which is correct differentiation.
Security posture
+ Validates token type. + UUID parsing prevents injection of non-UUID sub. – Does not check token audience/issuer/nonce; consider enforcing aud/iss when tokens are minted outside the API.
Performance
Decoding is O(1); DB hit only in strict dependency.
  • Good: explicit token-type check, structured errors with WWW-Authenticate, minimal payload usage.
  • Consider validating exp automatically (jose handles), plus nbf, iat skew handling via options if needed.
  • Optional auth returns None for invalid tokens (not only missing). This avoids leaking auth state, but can silently treat bad tokens as anonymous; ensure endpoints using it are tolerant.

2) Exception Handling (exception_handler.py)

Scope
Registers handlers for KonvertItError and broad Exception. Leaves HTTPException to default processing (good).
Logging
Uses logger.error for domain errors and logger.exception for 500s. Adds error_id correlation to 500 responses.
Mapping
Maps domain exceptions to appropriate HTTP codes (404/422/400/502/503).
  • Good: separation between domain and framework exceptions, correlation id for support.
  • Ensure PII is not in exc.message for user-facing detail. Consider adding request_id from structlog context to logs for better cross-correlation.

3) Logging (logging_middleware.py)

Correlation
Generates short request_id, binds via structlog.contextvars, includes X-Request-ID header. Clears context pre/post to avoid leaks.
Noise control
Skips /health, handles WebSocket upgrade bypass (BaseHTTPMiddleware limitation).
Metrics
Logs duration in ms, method, path, status, client IP.
  • Good: minimal and useful. Compatible with structured logging.
  • BaseHTTPMiddleware can evaluate body twice in some cases; ensure downstream handlers read body safely. Consider using ASGI middleware for zero-copy logging if profiling shows overhead.
  • Consider adding user id or auth subject when available (without PII) for better traceability; can be bound by router logic after auth.

4) Rate Limiting (rate_limiter.py)

Policy
Tier-based daily conversions; enterprise unlimited. Also tracks max_listings constant (not enforced here).
Storage
Redis with INCRBY and EXPIRE in pipeline; key namespace per user per UTC-day.
Headers
Returns X-RateLimit-* and Retry-After (on 429). Helper to inject headers.
Resilience
Fail-open on Redis errors with logging; returns unknown remaining (-1).
Observability
Warns when exceeding; pushes WS warning at ≥80% usage (best-effort, error-suppressed).
Security
Depends on get_current_user; uses user["sub"] as key. No IP-based fallback.
  • Good: atomic update with TTL; avoids double counting windows; enterprise short-circuit.
  • Fail-open is appropriate for UX, but consider feature flag or rate-limit shadow mode observability to detect abuse during Redis outages.
  • Singleton Redis client can survive across reloads in some servers; ensure shutdown hook calls close_redis().
  • Bulk limiter reads request.json(); on large bodies BaseHTTPMiddleware + reading JSON may increase memory; consider limiting max list size at router validation.

5) Security Headers (security_headers.py)

Headers
Sets X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Referrer-Policy, Permissions-Policy, CSP, optional HSTS in prod, and Cache-Control for API/health.
CSP
default-src 'none' and frame-ancestors 'none' fit an API-only backend. May block Swagger UI or embedded iframes if exposed.
Env awareness
HSTS only when settings.is_production.
  • Good: strong defaults for API services.
  • If serving interactive docs (Swagger/Redoc) from same app, CSP will likely block scripts/styles; consider conditional CSP relaxation for /docs and /redoc.

Actionable Recommendations

  1. Authentication:
    • Add issuer/audience validation to _decode_token when tokens originate from third-party IdPs. Example: pass audience and issuer via Settings and verify with jose options.
    • Optional auth: differentiate between missing token (None) and invalid token (log at info/warn, optionally return 401) depending on endpoint sensitivity.
    • Consider token replay protections if using long-lived access tokens (short access + refresh recommended).
  2. Logging and correlation:
    • Bind request_id into standard logger adapter so logs outside structlog also include it (logging MDC).
    • Include request_id in exception handler logs by reading from structlog.contextvars (if available).
    • Optionally log request content length and response length for performance tuning (avoid for sensitive endpoints).
  3. Rate limiting and resilience:
    • Add an application shutdown hook to invoke close_redis() to avoid descriptor leaks during reloads.
    • Enforce a max count for bulk URLs at pydantic schema level (e.g., 100) to prevent memory pressure.
    • Emit metrics (counter/gauge) for rate limit hits, near-limit warnings, and Redis errors.
    • Consider per-IP or per-route limits for unauthenticated endpoints, if any are exposed.
  4. Security headers and docs:
    • Gate CSP to allow Swagger/Redoc only on /docs and /redoc when enabled. Otherwise keep API-only CSP.
    • Consider adding Cross-Origin-Resource-Policy: same-origin for API-only deployments.
  5. Testing:
    • Existing tests appear to include test_auth_middleware.py, test_exception_handler.py, test_rate_limiter.py, test_security_headers.py. Ensure coverage includes Redis fail-open path and 80% WS notification branch (can mock ws_manager).
    • Add tests for optional auth returning None with invalid token and with wrong token type.
    • Test CSP behavior for /docs when enabled.

Configuration Assumptions

  • Settings.secret_key and jwt_algorithm are consistent with token issuer.
  • settings.redis_url points to a Redis with persistence not required for counters.
  • settings.is_production correctly reflects deployment stage.
