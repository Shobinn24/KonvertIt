KonvertIt Services & Tasks Audit

Scope

Reviewed files in app/services/ and app/tasks/:
  • services: billing_service, cache_service, compliance_service, conversion_service, discovery_service, price_monitor_service, profit_engine, sse_manager, user_service, ws_manager
  • tasks: task_queue, scrape_tasks, monitor_tasks, notification_tasks
Objective: evaluate architecture, reliability, performance, security, observability, and background processing patterns; provide actionable recommendations.

Summary

  • Strengths: clear service boundaries; strong error taxonomy; Stripe encapsulation; Redis cache fail-open; SSE + WS separation; conversion pipeline with callbacks; price monitoring with DB repos; structured search via ScraperAPI; DI-friendly factories/managers.
  • Medium risks: Stripe usage synchronous in async context; missing idempotency/verification on webhooks; potential resource leaks in long-lived workers; background tasks use in-memory singletons (non-multiworker-safe); rate pacing/limits not enforced for tasks.
  • High risks: None critical in code review. Ensure secrets/keys are handled via config and not logged.

Detailed Findings

BillingService

Scope
Customer creation, checkout/portal sessions, subscription status, Stripe webhook handlers (session completed, subscription updated/deleted, payment failed).
Errors
Custom BillingError; logs and maps Stripe exceptions; handles missing/invalid data.
Security
Relies on Settings for price IDs; no direct key leakage. Webhooks lack signature verification here (assumed handled at API layer).
Async
Uses stripe.* synchronous calls in async service methods → potential event-loop blocking.
  • Good: idempotent updates, tier mapping helpers, safe fallbacks.
  • Wrap Stripe SDK calls in run_in_executor or use an async-capable client to avoid blocking.
  • Add webhook signature verification and idempotency keys if not already at endpoint level.

CacheService

Behavior
Async Redis client with JSON serialization; fail-open on errors; namespace support; SCAN-based invalidation.
  • Good: resilient, safe defaults, TTL handling.
  • Consider compression for large payloads and metrics for hit/miss/error rates.

ComplianceService

Features
Loads VeRO brands; exact+fuzzy brand checks; restricted keyword scan; risk levels and violations.
  • Good: clear risk model; robust fallback on file errors.
  • Fuzzy threshold static; consider config-driven and caching for hot brands.

ConversionService

Pipeline
scrape → compliance → convert → price → list; step callbacks; error taxonomy mapping; bulk with cancel check and item callbacks.
Pricing
ProfitEngine used for suggest_price and breakdown; draft-only mode supported.
  • Good: strong orchestration with telemetry-friendly callbacks.
  • Serial bulk conversion; consider bounded concurrency with rate-limits per marketplace.
  • Surface more structured error data in result.error for frontend decisions.

DiscoveryService

Endpoints
ScraperAPI structured search for Amazon/Walmart; URL normalization for Amazon; pagination handling.
Errors
429 → RuntimeError; others via raise_for_status; skips ad URLs with logging.
  • Good: pragmatic normalization and robustness.
  • Add timeouts/jitter retry for transient 5xx; emit metrics for skipped items.

PriceMonitorService

Workflow
Scrapes current price for products with active listings; records history; updates product price; WS alert on change.
Errors
Per-product try/catch; records error in result.
  • Good: repository usage and WS notification.
  • Currently iterates products for users with active listings via user-wide product fetch; consider narrowing to products tied to active listings only to reduce load.
  • Add rate pacing and backoff for scrapes during sweeps; track sweep metrics.

ProfitEngine

Fees
Category-based fee table; payment fee and shipping; break-even and suggest price.
  • Good: explicit fee modeling and safe fallbacks.
  • Consider making fee tables configurable and updatable without code deploy.

SSEProgressManager

Model
In-memory job state and queues; heartbeat; formatted SSE events; cancel and cleanup helpers.
  • Good: clear event model and backpressure via queue.
  • In-memory store is single-process only; document limitation or add Redis-backed option for multi-worker scale.

WebSocketManager

Connections
Per-user connection lists; tier-based limits; broadcast; pruning on failure; singleton accessor.
  • Good: straightforward and testable.
  • Memory-only; add GC for idle connections and metrics (total, per-user) for observability.

Task Queue (arq)

Config
RedisSettings parsed from URL; worker settings include functions and startup/shutdown hooks; max_jobs=10, timeout=300s.
Infra
scrape_tasks maintains singleton Proxy/Browser managers and starts browser on startup; monitor_tasks schedules sweeps using DB session factory.
  • Good: resource reuse across tasks; graceful shutdown closes browser.
  • Redis URL parsing is minimal (no auth/ssl); consider using redis.from_url parsing or environment-specific overrides.
  • Long-lived Playwright contexts can leak on crashes; add liveness checks/restart strategy.

Actionable Recommendations

  1. Async/IO correctness:
    • Wrap Stripe SDK calls in an executor to avoid blocking the event loop or move to a sync worker context.
    • Add retries with backoff to DiscoveryService httpx requests for transient network/5xx errors.
  2. Webhooks and idempotency:
    • Verify Stripe webhook signatures and enforce idempotency at the endpoint; ensure BillingService handlers remain idempotent.
    • Record processed event IDs to avoid duplicate processing on retries.
  3. Background processing and resources:
    • Add health checks and periodic restarts for BrowserManager in workers; guard against zombie Playwright processes.
    • Bound concurrency in bulk conversions and monitor sweeps; add pacing tokens per marketplace.
  4. Data access efficiency:
    • In PriceMonitorService.check_all_for_user, join/filter to products behind active listings instead of fetching all user products.
    • Add pagination/batching for large product sets.
  5. Observability:
    • Add metrics: conversion steps, success/failure by cause, Stripe API timings/errors, cache hit/miss, SSE queue sizes, WS connection counts.
    • Propagate request_id into services when called in request scope.
  6. Configuration and security:
    • Externalize fee tables and compliance keyword lists to config or DB for runtime updates.
    • Harden Redis URL parsing (support redis://:password@host:port/db and rediss:// TLS).
  7. Testing:
    • Unit tests for BillingService handlers (updated/deleted) including tier mapping and idempotency.
    • Integration tests for ConversionService bulk with cancel and callbacks.
    • Load tests for SSE stream stability with heartbeats.
    • Tests for PriceMonitorService narrow queries and WS notification payloads.

Configuration Assumptions

  • Stripe keys and price IDs configured in Settings; webhooks routed to handlers with signature verification.
  • Redis accessible for cache and arq; Playwright dependencies installed on workers.
  • Single-process SSE/WS use acceptable for current deployment; plan for Redis pub/sub if scaling horizontally.
